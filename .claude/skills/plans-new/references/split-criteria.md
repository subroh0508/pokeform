# 計画分割の 6 基準（詳細・判断例）

`plans-new` が OVERVIEW を 1 phase = 1 PR に分割するときの判断材料。**規約の SoT は
[`.claude/rules/planning.md`](../../../rules/planning.md)** で、本ファイルはその 6 基準を運用しやすいよう
具体化した補助資料（progressive disclosure）。基準そのものを増減・変更するときは rule を直す。

## 大原則 — 1 PR = 1 つの完結した健全性の純改善

分割の狙いは「**レビュー可能で・単独でマージしても全体が壊れない・意味的に完結した単位**」に切ること. 大き
すぎる PR はレビューが形骸化し、小さすぎる PR は前後依存で文脈が散る。6 基準はそのバランスを多面的に
見るためのレンズであり、**どれか 1 つでも「重い」側に振れたら分割を検討**する。

## 6 基準

| # | 基準 | 「分割すべき」に傾くサイン | 「1 PR でよい」に傾くサイン |
|---|---|---|---|
| 1 | **意思決定の数** | 複数の独立した設計判断（型表現・データ表現・API 形）が混在 | 判断が 1 つ、または既存方針に従うだけ |
| 2 | **不可逆性** | 不可逆な決定（データ形式・公開 API・スキーマ）を複数含む | 可逆・局所的な変更のみ |
| 3 | **スコープの広さ** | 複数レイヤ（types / domain / io / cli）を横断 | 1〜2 ファイル・1 レイヤに収まる |
| 4 | **技術的難易度** | 難所が複数（新規アルゴリズム + 新規データ取得など） | 既知パターンの素直な適用 |
| 5 | **想定 diff** | ~500 行超が目安、**>1000 行は積極分割** | 数十〜数百行 |
| 6 | **並行実装のしやすさ** | 独立に並行で進められる塊に割れる | 直列依存が強く割っても待ちが多い |

## 想定 diff の扱い（基準 5 の補足）

- **~500 行**: 1 PR の快適なレビュー上限の目安。
- **>1000 行**: 原則**積極的に分割**する。レビューの質が落ち、回帰リスクが見えにくくなるため。
- **例外**: データセット追加（生成データの大量コミット等）のように、意味ある粒度での分割が**困難**な場合は
  1 PR を許容する。その際は**理由を OVERVIEW / issue に明記**する（「分割困難ゆえ 1 PR」と判断根拠を残す）。
  行数だけで機械的に切らない — 意味的完結性を優先する。

## 判断フロー（目安）

1. OVERVIEW の作業を列挙する。
2. 各作業を 6 基準で見て、**重い側に振れる基準が複数あれば分割候補**。
3. 分割するなら、基準 6（並行実装）と依存関係から phase の順序を決める（依存の浅いものを先に）。
4. 各 phase が「単独でマージしても壊れない・レビュー可能」かを確認する。崩れるなら切り直す。
5. すべて 1 つの完結単位に収まり、想定 diff も妥当（例外含む）なら **1 PR**（issue + implementation-workflow）。

## 判断例

### 例 A — 複数 phase に分割（MVP コア）

「個体・パーティの検証 + 一貫性/技範囲チェック + ステータス調整」をまとめて実装したい。

- 基準 1: 型表現・データ生成・CLI と独立判断が複数 → 重い。
- 基準 3: types / domain / io / cli を横断 → 重い。
- 基準 5: 合計で 1000 行超が見込まれる → 重い。
- 基準 6: 「データ生成」「検証層」「分析」は独立に進められる → 分割しやすい。

→ **複数 phase**（足場 + 計算 / データ生成 + チェック / 個体検証層 / 調整）。`docs/plan/NN-{slug}/` を起こす。

### 例 B — 1 PR 妥当（局所改修）

「相性表に 1 タイプの抜けを 1 件修正したい」。

- 基準 1〜4: 判断 1 つ・可逆・1 ファイル・既知パターン → すべて軽い。
- 基準 5: 数行 → 軽い。

→ **1 PR**。issue を起票して implementation-workflow へ。`docs/plan/` は作らない。

### 例 C — 行数は多いが 1 PR 許容（データセット追加）

「世代分のポケモン生成データを一括追加したい」。

- 基準 5: 1000 行超 → 通常は分割。
- しかし生成データは意味ある粒度で割りにくい（中途半端に分けても完結しない）→ **例外で 1 PR**。

→ **1 PR**。「分割困難ゆえ 1 PR」と理由を issue に明記して implementation-workflow へ。
