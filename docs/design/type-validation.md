# 型バリデーションの仕組み

pokeform は入力（個体・パーティ）の正しさを、**実行時のバリデーションライブラリではなく型検査だけ**で
保証する。この doc はその考え方と設計意図を説明する（型シグネチャ・具体的な型名の定義・閾値の実体は
持たない。正本は末尾「実装 SoT ポインタ」へ）。

## 唯一のゲートは「型チェック」

検証ゲートを **1 つ（型チェック）に絞る**のが核心の設計判断。入力は次の一方向の流れで検査する。

1. **入力（YAML/MD）を生成コードへ変換**する（コンパイル段）。この段は**常に成功してファイルを吐く**
   ——値の妥当性をこの段で例外として弾かない。
2. **生成コードを型チェッカにかける**。不正な入力は、生成コードが型エラーになる形で表現されているので、
   **型エラーとして検出**される。

この「不正を型エラーへ翻訳する」方針により、実行時バリデーションライブラリ（Zod 等）を**採用しない**。
検証器を別に持たず、型システムを唯一の真実源にすることで、オフライン・決定論的で、コーディング
エージェントが機械的に回せる検証を実現する。背景は [ADR 0010](../adr/0010-tsc-only-verification.md)。

## なぜ実行時バリデーションを採らないか

- **二重定義を避ける**: 実行時スキーマを別に持つと、型と検証ルールが二系統になりドリフトの温床になる。
  型を唯一のゲートにすれば、保証したい制約は型の上に一度だけ表現される。
- **依存を増やさない**: 実行時ライブラリを足さず、配布物を軽く・決定論的に保つ。
- **エージェント親和性**: 合否が型チェックの成否に一元化され、CI / エージェントが機械判定しやすい。

トレードオフは「型レベルでは重い制約（合計値のような算術）がある」こと。重い部分は変換段で先に計算して
型注釈へ逃がし、ゲート自体は型チェック 1 つに保つ（[[individuals-and-parties]] の合計値の扱いを参照）。

## 診断を読みやすくする — ブランドエラー型

型エラーは素のままだと「どの入力のどこが、なぜ不正か」が読み取りにくい。そこで、違反した箇所だけを
**意味の分かる名前を持つエラー型へ写像**する設計を採る。覚えない技・使えない特性・持てない持ち物・
合計値違反・同種族重複・未解禁といった各違反は、その意味を名前に含むエラー型として現れ、型チェッカの
メッセージに**違反の意味がそのまま出る**。これにより、診断を読んだ人やエージェントが原因へ最短でたどれる。

具体的なエラー型名・基盤となる名目型の仕組みは持たない（[[tsc-verification]] / [[type-conventions]] が SoT）。
決定の背景は [ADR 0019](../adr/0019-branded-error-types-and-source-mapping.md)。

## 生成コードから入力行への逆引き（`@source`）

型エラーは**生成コード上の位置**で出る。一方で利用者が直すべきは**元の入力ファイル（YAML/MD）の行**。
この距離を埋めるため、生成コードの各フィールドに**元入力の相対パスと行番号を指す印（`@source`）**を
埋めておき、型チェッカの診断位置を元入力の行へ逆引きして整形する。実行時バリデーションを採らなくても
「○○行目のこの技は覚えない」といった可読な診断を出せるのはこの仕組みによる。整形の責務は CL/IO 層
（[[cli-and-io]]）、写像の仕組みの SoT は [[tsc-verification]]。

## 種族粒度と ID キーの設計判断

型で制約を表すには、エンティティ（種族・技・タイプ・特性・持ち物）を一意なキーで指せる必要がある。

- **種族の粒度 = 種族値が一意に定まる単位**。フォルム・リージョン・メガで種族値が変わるものは別の種族
  として扱う。これにより「種族を指す」ことと「種族値・タイプを引く」ことが 1 対 1 になり、実数値計算や
  分析が取り違えなく行える。
- **キーは英名由来の安定 ID**（kebab-case）。改名されにくく型のキーに適するため英名 ID をキーにし、
  日本語名は別エンティティ（名前 SoT）に分離して持つ。入力を日本語名・英名のどちらで書いても、loader が
  名前 SoT から導出した照合表で正規の ID へ正規化してから検査する（[[type-conventions]] / [ADR 0035](../adr/0035-specs-languages-layout-redesign.md)）。
- **構造と名前を直交させる**: 種族値・タイプ・解禁などの構造値と、日英名は別々の正本に置く。混ぜないことで
  仕様変更時に直す場所が一意になる。メガは base 種族から構造を分離した独立エンティティとして持ち、base への
  逆参照で関係づける（[ADR 0036](../adr/0036-mega-independent-spec-entity.md)）。

## 巨大な ID 集合と「union を直接配らない」方針

種族・技などの ID 集合は 1000 種規模に膨らむ。これらを型レベルの大きな選択肢（union）として**直接配ると、
型チェッカの分配コストが重くなる**。そこで制約は、ID をキーにした**集約表へのプロパティアクセス主体**で
表現し、巨大 union をそのまま展開しない設計にする。性能懸念が出たらモジュール分割で対応する。

加えて、習得技などレギュレーションで変わりうる制約は**レギュレーションごと**に解決する。種族側に解禁
フラグを散らさず、レギュレーション側を一本の正本にして、レギュレーションを軸にした制約として表現する
（[ADR 0021](../adr/0021-per-regulation-species-and-legality.md)）。型表現の具体パターンは [[type-conventions]] が SoT。
