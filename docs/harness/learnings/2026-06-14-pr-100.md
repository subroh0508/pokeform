- **権限分離を agentType の非対称で構造的に担保**: 取得 = read-only（層2 の `Explore`・既存）/ 修正 = write（層3 の既定 agentType = Edit/Write/Bash 可・`model:"sonnet"`）。「パーサ修正とテスト追加はこの層だけ」をコードで表現。
- **無限ループ防止を三重に冗長化**: ① 同一 slug 修復試行 K 回上限（`maxRepair` 既定3）② `MAX_ROUNDS=K+1` ハード上限（各ラウンドで repairable 全 slug の試行を +1 するため証明可能に収束）③ 「1 グループも verify 緑にできなければ即停止」。収束性を設計で保証し、ライブ実走に依存しない。
- **同一パーサ欠陥をシグネチャ（stage + status + missingFields）でバッチ**し 1 修正 SubAgent に集約。「同じ欠陥が複数種で出る」前提（phase doc）をコスト効率よく満たす。

## ⚠️ Problem（詰まった点・制約）

- **層3 の自己修復ループをライブ実走で実証できていない**。実 fan-out + Sonnet 修復は実 DOM 揺れ + ネットワーク前提で、Phase 7（M-A 全データ投入）へ繰り延べた。収束性は設計（K 回上限 + ハード上限 + 進捗ゼロ停止）で担保するに留まり、`.workflow` は in-repo で機械検証できない（top-level return + 注入グローバルで tsc/biome 対象外）ため、ループ制御の論理ミスは実走まで検出されない（Phase 4 と同根の制約が層3 でも再発）。
- **`workflow({scriptPath}, args)`（Workflow のネスト呼び）プリミティブがリポジトリ内に前例・文書がない**。Workflow ツール仕様には存在するが、サポート前提が in-repo に明記されておらず、サポート外だと再 fan-out が不発になる（harness-review non-blocking 指摘 #1）。1 度の実機確認が望ましいが本 PR ではライブ実走を繰り延べたため未確認。
- **修正 SubAgent が main worktree の作業ツリーを直接書き換える設計**で、複数修復グループを parallel 起動すると同一ファイル（`parse.ts` 等）への競合が起こりうる。本 PR はバッチ集約（欠陥シグネチャ単位）で並列数を抑えるが、worktree 分離（`isolation:"worktree"`）は使っていない。実走時の競合は未検証。
- **finish-phase の README 進捗更新が feature PR に同梱されず learning ブランチへ相乗り**（Phase 4 と同じくオーケストレーター主導マージで feature マージ後に進捗更新が確定）。反復している運用パターン。
- **層3 が層2 の返り値 shape（`{ok, failed, counts}`）に密結合**。層2 の契約が変わると層3 が静かに壊れるが、`.workflow` 同士の契約はテストで固定できない（両者ゲート対象外）。
