- **spec 正本のパス表記が PR スコープ外に取り残されかけた**: 主旨が「3 ファイルへの最小追従」だったため、`docs/plan/01-mvp/architecture.md:261,263,308` の旧 `regulations/<id>.yaml` 表記が当初差分外。harness-review の nit で気づき同一 PR で拾えたが、レイアウト移行 PR は「全 SoT のパス表記」を初めから追従対象に含めないと取りこぼしやすい。
- **`indexOf("-")` による game/reg 分割が脆い**: `serebii-to-catalog.ts` の `regId.indexOf("-")` 分割は、ゲーム省略形 `m-a` に対し `m/a.yaml` という意図しない第1候補を生む。`existsSync` で弾かれるため現状無害だが、将来ゲーム名にハイフンが入ると分割が曖昧になる潜在債務。
- **列挙順がグルーピングで変わる潜在差分**: `generate.ts` の列挙が flat sort から game→file の 2 段ソートへ変わり、**複数ゲーム存在時**に生成 index の挿入順が旧実装と変わりうる。単一ゲーム `champions` のみの現状は diff 空で無害だが、多ゲーム化時に顕在化しうる（id ソートが効いていれば実害なし）。
- **basename 表示の同名衝突リスク**: `check-regulation.ts:83` の出力 `name` は basename 切り詰めのため、別ゲームに同名 `m-a.yaml` が存在すると表示が衝突しうる（検証ロジックは非影響・表示のみ）。
- **per-game レイアウトの「なぜ」が本 PR で未 ADR 化**: グルーピングは可逆なレイアウト変更として ADR を見送り、決定の背景は後続 Phase（per-game 技メタ化）へ先送り。妥当な判断だが、レイアウト方針の根拠が一時的に PR body にしか残らない区間が生じる。
