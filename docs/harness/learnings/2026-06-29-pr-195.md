## ⚠️ Problem（詰まった点・制約）

- **cheerio が table 外の `<td>` を黙って落とす**。decode helper の inline test を `<td>...` だけで書いたら `$("td")` が空になりテストが偽陰性化。`<table><tr>` で必ずラップする必要があった（latin-1 + 非ブラウザ DOM の落とし穴）。
- **latin-1 fixture を grep するとロケールでヒットしない**。`LC_ALL=C` を付けないと `<title>` すら拾えず、構造確認に手間取った。バイナリ寄り fixture の調査は最初から `LC_ALL=C` 前提にすべき。
- **mega の ja が構造的に取得不能**。Serebii 種族ページの mega `<h3>` は英語（"Mega Charizard X"）でカナを持たず、速報の主目的「ja 抽出」が mega だけ no-op。空 ja を書くと net-new で generate が fail-fast するため、sync で空 ja を落とす後追い修正が必要になった（code-review 指摘）。
- **items.shtml から megaSpecies（X/Y 枝）を判別できない**。sync-showdown は megaStoneFor+megaSpecies 両方書くが、Serebii items 経路は base 種族しか分からず megaSpecies を書けない。net-new メガストーンで generate fail-fast の構造的死角が残る（速報の限界として許容・showdown が authoritative）。
- **ADR archive 退避の inbound 追従が広範**。0033/0037 への clickable リンクが ADR 本文 / README / design / rules / completed plan / survey-regulation skill に散在し、相対深度が階層ごとに違う（`../adr/` / `../../../adr/` / `../../docs/adr/`）。Phase 5 で改訂・削除予定の rule/skill にも追従が必要で、「dangling ゼロ」と「内容改訂は後続 phase」の境界判断を要した。
- **`peter-evans/create-pull-request` 生成 PR の CI は別途回る前提**で、ワークフロー内 inline verify が正経路（showdown-sync 既存と同型）。実機 dry-run はこの環境で workflow_dispatch を起動できないため、YAML 構文 + ローカル sync スモークでの代替検証に留まった。
