## ✅ Keep（継続したいこと）

- **値不変の機械検証を AC に明記し実際に裏取りした**: 「所在移動のみ・値不変」を、生成物 Before/After の一字一句一致 + 再生成で追加差分なし（決定論的）+ runtime `regulationDex[R].name` 解決で確認。所在移動 refactor で「値が変わっていない」を曖昧にせず証明した。
- **既存ヘルパ（`requireNames` / `acc` / `lit` / `langFiles`）の再利用で新機構を足さなかった**: レギュ名専用の emit 機構を作らず、`langFiles` 配列に 1 行追加するだけで languages dex 出力・index 集約・突き合わせゲートに乗せた（over-engineering 回避）。
- **由来コメントで「なぜ languages 由来か」を ADR 0035 へポインタした**: `generate.ts` の name 合成箇所・`regulation.ts` の `name` フィールド・`RESERVED_REGULATION_KEYS` 除去箇所・check-regulation の各所に理由コメントを残し、後続が背景を辿れる。
- **予約キー除去とレコード構築の両側を整合させた**: `RESERVED_REGULATION_KEYS` から `name` を外すだけでなく `loadRegRecord` から `name` を落とし、片側だけ変えると `bad-species-block` 誤検出になる罠を回避（テストが先に検出）。
- **code-review の nit（doc-data 乖離）を phase スコープ内として即追従した**: code-review が「スコープ外」とした data-pipeline.md の例外記述を、phase doc「育てるハーネス」節の射程と判断し同 PR で解消、harness-review でも独立検証した。

## ⚠️ Problem（詰まった点・制約）

- **予約キー除去が既存テスト fixture を壊した**: `RESERVED_REGULATION_KEYS` から `name` を外した結果、fixture 内の `name:` キーが種族ブロック扱いになり `bad-species-block` 誤検出で verify 赤。fixture 追従が必要と事前に洗い出せず、verify で初めて顕在化した。
- **README 進捗 checkbox が phase PR に含まれず別 PR（retro）へ持ち越し**: Phase 4 も同様に未チェックのまま merge されており、phase PR 内で進捗更新する運用と retro/orchestrator が一括更新する運用が混在している。
- **ハーネス資産変更が後追いで PR に混入しレビュー往復が増えた**: 最初 code-review（src/scripts 専用）→ nit 指摘で rule 修正 → harness-review、と 2 段になった。レギュ名移動が rule 記述に波及することは着手時に予見でき、最初から rule 追従を含められた。
- **ADR 0035 本文がレギュ名を明示列挙していない**: 「名前 SoT を languages へ」の射程にレギュ名が含まれることは phase doc の判断に依存し、ADR 本文だけ読むと射程が一見曖昧（追補不要の判断は妥当だが本文からの自明性は低い）。
