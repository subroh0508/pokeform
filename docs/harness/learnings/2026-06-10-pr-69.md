- **lossless 移行を機械抽出で実現**: 既存名を現生成物（`data/generated/*.ts`）から抽出して catalog YAML へ書き戻し、regenerate 後の生成差分が `abilities.ts`/`items.ts` の name 除去のみに収束（types/moves/names/species-base/regulations はバイト不変）。移行の正しさを差分ゼロで裏取りできた。
- **一度きりの移行スクリプト + 削除**: moves 219 等の大量名を手書きせず決定論的に移行し、完了後にスクリプトを削除（生成物→入力への逆流を最小の使い捨てで済ませた）。
- **authoring ゲートの実挙動確認**: ja/en を 1 件欠落させ `generate:data` が非0終了することを実際に確認（コードを書くだけで満足しない）。
- **doc-data 同期を同一 PR で**: rule（data-pipeline / type-conventions）・architecture.md・ADR 0025・survey-regulation skill を実装と同じ PR で同期し、ドリフトを残さなかった（Phase 4 セルフ検証）。
- **独立 2 レビューの分担が機能**: code-review が lossless を実機 regenerate で裏取り、harness-review が reference ファイルの取りこぼしを検出（Generator/Evaluator 分離 + src/harness 分担）。

## ⚠️ Problem（詰まった点・制約）

- **catalog 消費者の grep 漏れ**: `join(CH, "catalog", file)` は文字列 `catalog/` に当たらず、初回調査で `check-regulation.ts` の `readCatalogSet` を見落とした。`pnpm verify`（型/テスト/Biome）は緑でも、**pre-commit フックの `check:regulation`**（catalog をリスト前提で `new Set(list)`）で初めて `object is not iterable` で発覚。verify とフックのゲート差で遅れて気づいた。
- **skill 本体と reference のドリフト**: `survey-regulation/SKILL.md` を新形式へ更新したが `references/serebii-sourcing.md` の旧 `itemMeta` 記法を取りこぼし（harness-review が検出・diff にも含まれず）。
- **data/raw 非共有**: ローカル `data/raw`（gitignore）が空で、generate 前に `fetch:data`（network）が必要だった（checkout/worktree で raw 共有されない）。
- **移行はデータ品質を改善しない**: 既存の疑わしい ja 名（`aspear-berry` 等）が lossless 移行で温存され、名前手書き化後も別途の修正が要る。
