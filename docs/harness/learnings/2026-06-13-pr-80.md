## ⚠️ Problem（詰まった点・制約）

<!-- 最低 3 / 推奨 5-10 -->

- **renumber の機械置換が「Phase N / phase-N」表記前提で、文中の素の番号を取りこぼす**: OVERVIEW の narrative には M-A を指す素の「16」（例 `全量投入（16）`・`16 を de-risk`）が多数あり、`Phase 16`/`phase-16` の sed では掛からず、別途 perl で逐一置換する手作業が必要だった。renumber の番号整合は 3 形態（`Phase N` / `phase-N` / 文中の素番号）を意識しないと漏れる。
- **前回 renumber（PR #76）の取りこぼしが stale 参照として残存していた**: OVERVIEW 末尾の `- 15 はデータセット追加で…1 PR 許容` は M-A 投入（当時 16）を指す根拠記述だが、PR #76 の `15→16` renumber 時に bare number `15` が更新されず残っていた。renumber 漏れが累積し、今回 `20` へ直して初めて解消した。素番号の取りこぼしが恒常的なリスクであることの証跡。
- **markdown フォーマッタ hook 直後の Edit が "modified since read" で複数回弾かれた**: 編集 → hook 整形 → 次の Edit が古い state を参照してエラー、再 Read のオーバーヘッドが複数回発生した。
- **`grep "X" 0 件` 系の受け入れ基準が自己言及 doc と衝突しうる**: phase-18 の AC「`hand-authored` が 0 件」は、rename を論じる phase doc 自身が同語を含むため棚卸し対象になり得る（reviewer が non-blocking 指摘）。AC の「意図的残存は理由を doc 化」逃げ道に依存している。
- **skill 境界（merge しない）とユーザー明示指示（merge して）のギャップ**: pr-retrospective も implementation-workflow の一部段階も「skill 自体は merge しない / 人間 approve 後」を原則とするが、ユーザーが明示で「マージまで持っていって」と指示した。orchestrator がユーザー指示を根拠に merge する運用例外であることが skill には明記されていない。
