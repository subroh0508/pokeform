> 構造データ（種族値 / タイプ / 特性 / 全国図鑑番号 / 持ち物 category）の SoT を `data/raw` 直読から `data/champions/catalog/*.yaml` へ移し、`generate.ts` を raw 非依存化。raw→catalog 転記専任の `materialize.ts` を新設し、`overrides.yaml` を廃止して ADR 0027 / skill / rule を追従させた（Phase 13）。

## ✅ Keep（継続したいこと）

<!-- 最低 3 / 推奨 5-10 -->

- **phase doc の検証手順を実機で回して de-risk**: raw 退避 → `generate:data` 等価生成・raw 不在 → `materialize` 非0終了（ENOENT fail-fast）を実際に実行して確認できた。検証 1〜5 を順に回す手順設計が回帰リスクを小さく抑えた。
- **純関数と I/O の層分離**: 転記計画の純関数を `src/codegen/materialize.ts` に、fs/YAML I/O を `scripts/materialize.ts` に分離。純関数だけを単体テスト可能にし、`extractSpeciesStructural` / `extractItemCategory` / `planFields` を全分岐網羅でテストできた。
- **生成物 byte 等価を移行のゴールに据えた**: `generate.ts` を raw 非依存化しつつ `git diff data/generated` 空を受け入れ基準に置いたことで、SoT 移設という大きな構造変更でも回帰ゼロを保証できた。
- **責務の二重化を避ける設計判断**: raw 存在担保を `survey-regulation` skill 責務に寄せ、`materialize.ts` は fail-fast（ENOENT 即終了）のみに絞った。存在チェックと `fetch:data` 誘導をスクリプトに持たせない切り分けが明快。
- **hand-authored 修正の保護**: `planFields` が未設定フィールドのみ fill し既存値は conflict 提示（上書きしない）設計で、Champions 実態に合わせた手修正が raw 再転記で潰れない。`parseDocument` でコメントも保持。
- **独立レビュー（code/harness 分離）が機能**: source スコープと harness スコープを別エージェントで起動し、両者とも blocking 0。catalog 値スポット照合・rules-index 再生成等価・ADR 採番規約準拠まで具体的に検証された。
- **ADR で SoT 移設の不可逆判断を記録**: ADR 0027 を起票し「取得元 = PokeAPI 維持」「ADR 0025 の構造データ版拡張」を明記。ADR 0012 を supersede しない切り分けを残した。

## ⚠️ Problem（詰まった点・制約）

<!-- 最低 3 / 推奨 5-10 -->

- **`acceptEdits` でも Bash 都度承認が多発**: worktree + 別ペインのワーカー駆動で、編集は自動承認でも Bash コマンドが都度承認を要求し反復ロスが発生。PR #72 と同様の反復課題で、オーケストレーション体験を阻害した。
- **catalog のヘッダコメントが SoT 移設に追従していなかった**: `species.yaml` / `items.yaml` の冒頭コメントが旧「data/raw 由来」のまま残り、ADR 0027 の SoT 移設と矛盾。次の編集者が raw を直しに行く事故を招く記述で、レビュー指摘を受けて修正コミットを追加した。
- **テスト対象の純関数がカバレッジ除外パスに同居**: `extractSpeciesStructural` / `planFields` は stat マッピング・conflict 検出という意味あるロジックを持つが、`src/codegen/**` は coverage 除外（薄い層）扱い。コロケーション test はあるが分岐が 100% ゲートで強制されない構造的ギャップ（code-review non-blocking）。
- **ADR 本文の rule 参照記法が混在**: ADR 0027 本文の `[[data-pipeline]]` 等 wikilink 表記が、他 ADR の相対 markdown リンク主体と不揃い（harness-review nit）。実害はないが ADR 内のリンク記法に明文の統一規約がない。
- **overrides 廃止の波及範囲が広く追従漏れリスクが高かった**: AGENTS.md / architecture.md（mermaid ノード含む）/ code-review skill + checklist / data-pipeline rule と多数の資産に overrides 記述が散在。grep で active/narrative を切り分けたが、追従対象が手作業で広範だった。

## 🚀 Try（次回の改善案）
