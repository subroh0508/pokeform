## ⚠️ Problem（詰まった点・制約）

- **実装中の API 不安定で worker ターンが複数回 idle 化**した（ECONNRESET / 壊れた tool 呼び出し）。長尺タスクが途中で停止し、orchestrator が commit → PR → code-review → merge を代行する事態になった。worker 単独での端から端までの完走が崩れた。
- **biome override が `linter.enabled:false` のみで formatter は `src/generated/**` に依然適用**される（nit）。PR 概要の「Biome から除外」は厳密には「lint のみ除外・format は残る」で、文言と実体に乖離があった。生成物が biome 整形と等価なので実害はないが、概要記述の精度に課題。
- **`package.json` の exports/files glob 追従**を PR 概要が挙げたが、実体は `"private": true` で `exports`/`files` を持たず該当設定が無かった。概要が存在しない対象を挙げており、事前のファイル実在確認が甘かった。
- **62 files の大規模 diff** は git mv 主体とはいえレビュー負荷が高い。生成物 22 ファイル + consumer + config + doc が一括で、レビュアは「rename か実質変更か」を 1 件ずつ判別する必要があった。
- **ADR / 過去 learning / 旧 phase doc の `data/generated` 参照**は不変ログ・point-in-time 記録として意図的残置だが、`git grep "data/generated"` の残存ゼロ判定で「移行注記を除く」例外を都度判断する必要があり、追従対象と非対象の線引きが手作業だった。

## 🚀 Try（次回の改善案）

- **PR 概要の「除外」記述を機構粒度で書く規約**: 「Biome から除外」のような粗い表現でなく「Biome lint のみ除外・format は継続」「カバレッジ除外・型検査は継続」と機構（lint/format/coverage/typecheck）ごとに明示する。`code-review` skill の checklist に「概要の除外記述が機構粒度か」を点検観点として追加検討。
- **長尺タスクの API 障害時の代行手順を rule 化**: worker が idle 化したとき orchestrator が commit → PR → review → merge を引き継ぐ手順を `implementation-workflow` rule / skill に明文化。中断してもディスクの commit が保全される前提（worktree 分離）を活かした再開・代行プロトコルを SoT 化する。
- **PR 概要に挙げる変更対象の実在を事前確認する運用**: `package.json` exports/files のような「触る予定の設定」が実在するかを概要記述前に grep 確認する。概要が存在しない対象を挙げると review で空振り点検が発生する。
- **生成物 formatter 除外の判断を data-pipeline rule に注記**: 生成物が biome 整形と等価な現状は `formatter.enabled:false` 不要だが、emit テンプレートが整形等価を崩すと drift する。「生成物は lint 除外・format は決定論等価で維持」の判断根拠を [[data-pipeline]] に残す選択肢。
