## ⚠️ Problem（詰まった点・制約）

- **Phase 1 の 3 ギャップが M-B でも全て再現**: ① Champions 固有メガ（9 形）が PokeAPI 非解決で materialize 不可 / ② 新規種族 null placeholder で materialize クラッシュ（21 種 + 14 持ち物を `{}` へ手動変換）/ ③ mega ja・ability が手動著述依存。Phase 1 で「申し送り」にしたまま未是正のため、限定投入のたびに同じ手作業が増える。
- **null placeholder 起因の megaEvolvesTo 自動著述スキップ（新規発見）**: `writeSpeciesCatalog` の `"set" in node` 判定が null Scalar placeholder で false になり、新規 base 種（sceptile/blaziken/swampert/mawile）の `species-specs.megaEvolvesTo` が書かれない。既存 entry の metagross のみ著述された。Phase 1 では既存メガ種が catalog 在籍だったため顕在化しなかった。
- **gendered 種の PokeAPI form-slug 不一致（新規発見）**: `pyroar` は PokeAPI の default `pokemon` が `pyroar-male`（`pokemon/pyroar` は 404）。`fetch:data` に slug override が無く、`pokemon/pyroar-male` を raw キャッシュに `pyroar` 名で保存する手動 fallback を要した。本投入では meowstic/indeedee/basculegion 等の gendered 種で同問題が多発しうる。
- **reg dir の scaffold 手段が無い**: 空の m-b/* を手で初期化（`species: []` / `mega: {}`）し、materialize 後に flow→block 正規化パスを自前で書く必要があった。新レギュ着手のたびに同じ定型作業が要る。
- **`{}` placeholder workaround が flow style を残す**: materialize が `{}` を埋めた結果 flow style になり、`check:yaml-style` を通すため block 正規化を追加で要した。null/`{}`/flow の三つ巴が毎回手間。

## 🚀 Try（次回の改善案）

- **materialize を null/`{}` placeholder 許容にする**（Phase 3 の本丸）: `node.toJS()` が null/空を返しても planFields がクラッシュせず構造を埋め、block style で書き戻す。これで null→`{}`→正規化の手作業 3 連が消える。
- **megaEvolvesTo 自動著述を placeholder にも適用**: `writeSpeciesCatalog` で null placeholder を先に空 block map 化してから megaEvolvesTo を set する（`"set" in node` 判定の前提を満たす）。
- **Champions 固有メガ/アイテムを Serebii scrape 由来で著述する経路を skill に追加**: PokeAPI 非依存で mega の dex/types/stats/ability/ja を scrape JSON から転記し、固有メガ（Mega Staraptor 系・Mega Starmie/Dragonite）を本投入で起こせるようにする。
- **`fetch:data` に gendered/form-slug override マップを追加**: `pyroar→pyroar-male` 等の data 駆動マッピングで、gendered 種の 404 を自動解決する（rotom 系 form-slug も同経路で吸収できるか検討）。
- **reg dir scaffold コマンドを追加**: `survey-regulation` に空 reg ディレクトリ（block style 初期構造 + index period）を起こすサブコマンド/手順を追加し、手動初期化と正規化パスを定型化する。
