> 02-data-model-redesign Phase 1。roster.yaml を catalog/{species,moves,items,abilities}.yaml の append-only マスターへ分離。

> 種族 abilities のカタログ参照整合を generate.ts で throw 化。code-review / harness-review とも blocking 0。

## ✅ Keep（継続したいこと）

- **生成物バイト等価で構造変更を達成**: abilities カタログを現状の自動収集集合で seed・同順ソートしたため `data/generated/*` の git diff が差分ゼロ。構造の付け替えと出力不変を両立し、レビューが裏取りしやすかった。
- **参照整合を生成段の throw で担保**: 種族の特性がカタログに無いと `generate.ts` がエラー（実証: rough-skin を削ると `ability 'rough-skin' of 'garchomp' not in catalog/abilities.yaml`）。append-only マスターの整合を機械的に守れる。
- **doc-data 同期を同一 PR で実施**: roster→catalog の変更に合わせ architecture.md（正本）/ data-pipeline.md（rule）/ regulation.yaml コメントを同期。Phase 4 セルフ検証の「rule が正本に先行する乖離」を未然に防いだ。
- **worktree 物理分離で実装**: implementation-workflow Phase 0 に従い worktree で隔離実装。main を汚さず並行性を確保。
- **独立レビューを Generator/Evaluator 分離で実施**: ソースは code-review、ハーネス資産は harness-review を別サブエージェントで並列起動。両者が独立に等価性・参照整合を再現確認した。
- **append-only 方針をコメントで明文化**: 各 catalog YAML 冒頭に「没収されても消さない」方針と vendor スコープの役割を記載し、後続フェーズの誤削除を予防。

## ⚠️ Problem（詰まった点・制約）

- **roster 参照の追従漏れ**: phase doc 自身が「grep で洗う」と注意喚起したのに、初回 grep を `head -40` で打ち切り `author-individual/SKILL.md` の旧 roster.yaml 参照 2 箇所を見落とした。harness-review が non-blocking で検出し fix loop 1 回を要した。
- **worktree 間で data/raw 非共有**: gitignore の raw キャッシュは worktree に無く、generate がコケた。メイン側にも raw が無く、結局 fetch:data（ネットワーク）が必要だった。worktree 実装時の raw 取り回しが定型化されていない。
- **catalog コメントの前方参照**: 未作成の `data/champions/regulations/*.yaml`（Phase 2 成果物）を具体パスで先行参照し、両レビューが nit 指摘。具体パスなしへ緩和した。
- **生成物等価の確認が手作業**: 「バイト等価」を git diff で都度確認する手順が暗黙。等価維持が要件の PR で機械的に保証する仕組みが無い。
