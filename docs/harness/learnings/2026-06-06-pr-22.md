- **implementation-workflow Phase 0〜6 のワーカー駆動 + 別ペイン・オーケストレーターのマージ分担**が機能した。worktree 物理分離で main を汚さず実装でき、マージ衝突なく #22/#23 が順に取り込まれた。

- **Generator/Evaluator 独立**: code-review を別サブエージェントで起動し PR コメントに投稿。自己採点を避けられ、nit（整数性検証）も第三者視点で拾えた。

## ⚠️ Problem（詰まった点・制約）

- **main が保護ブランチ**で、finish-phase の README 進捗更新を直接 push できず別 docs PR（#23）化が必要だった。implementation-workflow / finish-phase の手順にこの分岐が明記されておらず、一度直接コミット → reset → ブランチ切り直しの手戻りが発生した。
- **worktree は node_modules を共有せず**、Phase 0 直後に `pnpm install` が必須。手順に明記がなく、最初の verify が失敗しかけた。
- **phase doc「この Phase で育てるハーネス」の `new-domain-fn` skill** がタスク `[ ]` 一覧の外に記載され、src 集約 PR のスコープ内か harness PR へ分離かの判断が曖昧だった（今回は deferred 判断）。
- **Node v23 環境で `engines: >=24` の `[WARN] Unsupported engine`** が全 pnpm コマンドに出続けノイズになった（無害だが verify ログの可読性を下げる）。
- **isValidPointAllocation が整数性を検証しない**（nit）。`PointValue` 型で守れない外部入力（YAML 由来の生値）向け保険という doc コメントの趣旨に対し、`0.5` 等の非整数が合計 66 に収まると通過し得る。
