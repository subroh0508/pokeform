## ✅ Keep（継続したいこと）

- **層分離でカバレッジ 100% を素直に達成**: 判断ロジックを `src/domain/*` の純関数（type-effectiveness / coverage / resolve-names / party-analysis）に集約し、`io`/`cli` を薄い glue として明示除外。「除外で取り繕う」でなく層の性質による除外で 100% を維持できた。
- **値→型の単一ソース化**: `export const xxxDex = {...} as const` から `type XxxDex = typeof xxxDex` / `XxxId = keyof XxxDex` を派生し、型と値の二重管理を排除。親型適合は `satisfies` / `Assignable` で tsc 検証。
- **vendor スコープを `roster.yaml` で宣言的に制御**: 取得対象を 1 ファイルに集約し、`fetch-pokeapi.ts` は raw キャッシュ固定で決定論的、`generate.ts` はネットワーク非依存で再生成可能。スコープ拡張が宣言の変更だけで済む。
- **ja/en 混在サンプルで lang 厳格化を実証**: `team/` に ja 個体・en 個体・脆弱デモ・不整合デモを用意し、受け入れ基準（弱点集中の非0終了・重複/未解禁/参照切れ検出）を実コマンドで確認できた。

## ⚠️ Problem（詰まった点・制約）

- **全種族の literal 型生成が非現実的**: 1025 種 × 全習得技を literal 型で materialize すると tsc 負荷・取得時間・ファイル肥大が過大。MVP は代表種サブセット（8 種）+ 全 18 タイプへスコープ調整した（architecture の「全種族生成」と差分が発生）。
- **native TS 実行と相対 import の拡張子**: `node src/cli/index.ts` 実行は相対 import に `.ts` 拡張子を要求する一方、tsc の bundler resolution は拡張子なしを許す。`allowImportingTsExtensions` 有効化 + 全相対 import への `.ts` 付与（codemod）で両立させたが、着手時に気付けず後追いになった。
- **派生 Id の自己参照による型循環**: `SpeciesBase.megaEvolvesTo: SpeciesId`（= `keyof typeof speciesDex`）を inline `satisfies` で検証すると `typeof speciesDex` 計算が自己参照して循環。`as const` で型を独立させ `Assignable<Record<string, XxxBase>, XxxDex>` を分離する回避が必要だった。
- **architecture.md の SoT ドリフト**: 型表現パターンの実装（`interface XxxDex` 手書き → `typeof` 派生 / `species.ts`+`species.data.ts` 分割解消）に対し正本 architecture.md が未追従（本 follow-up PR で解消）。harness-review が「早期解消推奨」と指摘。
- **weak type による代入拒否**: 全フィールド任意の `ItemInfo` が weak type となり生成 `itemDex` を弾いた（共通プロパティ無し）。識別子 `id` 必須化で回避したが、最小情報型の設計に落とし穴があった。
- **診断 UX の弱さ**: `regulation` 未指定パーティで全メンバーが `not-legal` 誤爆する（code-review 指摘）。「レギュ未指定」と「未解禁」を区別する診断が望ましい。
