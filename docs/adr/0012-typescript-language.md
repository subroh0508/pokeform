# 0012. プログラミング言語に TypeScript を採用する

## Context

pokeform は「Pokemon as Code」= 育成済み個体やパーティを**コードとして表現し、コーディングエージェントが機械的に検証**しながらブラッシュアップする土台。中核要件は (1) 型を**検証ゲートそのもの**にできること（[ADR 0002](./0002-tsc-only-verification.md): `tsc --noEmit` を単一ゲートにする）、(2) 種族・技などのデータを**型として表現**できること（[ADR 0003](./0003-species-dex-pattern.md): `XxxDex` パターン / リテラル型 / テンプレートリテラル型 / ブランドエラー型）、(3) npm エコシステムとエージェントの親和性。言語選定はこれらを満たすかで決まる。具体バージョンには言及しない（版の SoT は [ADR 0017](./0017-toolchain-version-source-of-truth.md)）。

## Decision

実装言語に **TypeScript** を採用する。理由は、静的な構造的型システムを**検証ゲート兼データ表現手段**として使え、かつ npm エコシステム上でエージェントが扱いやすい唯一現実的な選択肢だから。リテラル型・テンプレートリテラル型・mapped/conditional 型により「データを型として表現し、不正を型エラーで検出する」設計（[ADR 0002] / [ADR 0003]）が軽量に実現できる。

## Consequences

- **良い点**: 型を検証ゲートに転用でき（実行時バリデーション不要）、データ整合性をコンパイル時に保証できる。npm/Node エコシステムとエージェント支援が厚い。
- **悪い点 / コスト**: 型システムは構造的ゆえ実行時の値までは保証しない（境界は別途設計が要る）。重い型レベル計算はコンパイル時間に影響する。
- **トレードオフ / 留意点**: 「型＝ゲート」を活かすため、言語機能（リテラル/テンプレート/conditional 型）に依存した設計になる。

## Alternatives Considered

言語選定として、動的型付け（JS 以外）と静的型付けの代表を比較した。

| 言語 | 型付け | 却下理由 |
|---|---|---|
| **TypeScript** | 静的・構造的 | ✅ 採用。型を検証ゲート + データ表現に転用でき、npm エコシステム/エージェント親和 |
| JavaScript（素の JS） | 動的 | コンパイル時の型検証が無く、型をゲートにできない（[ADR 0002] の単一ゲート前提が成立しない） |
| Python / Ruby | 動的 | 同上。型ヒント/型注釈はあっても、不正データを「型エラー」として機械検出する設計に乗らない。実行時バリデーションは [ADR 0002] で却下 |
| Go | 静的・名前的 | 静的型はあるが npm エコシステム外。リテラル型/テンプレートリテラル型が無く、`XxxDex`・ブランドエラー型（[ADR 0003]）相当を型で表現しにくい |
| Rust | 静的・名前的 | 型表現力は高いが学習/ビルドコストが重く npm 外。データ-as-types を軽量に書く用途には過剰 |
| Kotlin / Java | 静的・名前的 | JVM/エコシステムが別。構造的型・リテラル型による data-as-types 表現が TS ほど軽量でない |

要点: 「**型をそのまま検証ゲートにし、データを型として表現する**」要件を、npm エコシステム上で軽量に満たせるのは TypeScript だけ。動的型付け言語は型ゲートを持てず、他の静的型付け言語はエコシステム外かつ data-as-types 表現の軽さで劣る。
