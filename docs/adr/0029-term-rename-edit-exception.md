# 0029. 用語 rename を ADR 本文の整備例外として許可する

## Context

[`.claude/rules/adr.md`](../../.claude/rules/adr.md) は「Accepted の ADR は本文を書き換えない」を不変ログの原則とし、例外は**可変な plan ファイル参照（phase doc / 「Phase N」表記 / `OVERVIEW.md` リンク）の遡及除去**という整備だけを許してきた。これは「決定の本質を変えない機械的な書き換えなら整備として許す」という線引きを最初に確立した前例である。

その後、ハーネス内の用語を平易な日本語へ統一する必要が生じ、既存の Accepted な ADR 本文に含まれる同一概念の英語表記を別表記へ機械的に置き換えたい場面が出てきた。これは決定の内容・適用範囲を一切変えない純粋な表記の言い換えだが、現行の `adr.md` 文面では plan 参照除去以外の本文編集が許されず、用語統一が原則と衝突して宙に浮く。plan 参照除去で確立した「意味を保存する機械的整備は許す」論理は、用語 rename にもそのまま延長できる。

## Decision

**決定の本質を変えない用語 rename（表記の言い換え）を、可変 plan 参照の遡及除去と並ぶ ADR 本文の整備例外として許可する。** 許可範囲は[`.claude/rules/adr.md`](../../.claude/rules/adr.md)（「Status と supersede」節）に定める線引きに従う:

- **許可**: 機械的・意味保存の言い換えに限る（同一指示対象を別表記に置換するだけで、決定の内容・適用範囲が一切変わらないもの）。逐一どの語をどう置換したか確認できる単位で行う。
- **非許可**: 決定文・理由・Consequences・Alternatives の意味を変える書き換え。これは整備ではなく決定の見直しであり、引き続き禁止する（覆すときは supersede = 新 ADR + 旧 ADR の `status` 更新 + archive 退避による）。

## Consequences

- **良い点**:
  - ハーネス全体の用語を統一でき、Accepted な ADR だけが古い表記で取り残される doc ドリフトを防げる。
  - 「意味を保存する機械的整備は許す」という既存の線引き（plan 参照除去）を一貫した形で延長でき、整備例外の判断基準が 1 つに揃う。
- **悪い点 / コスト**:
  - 「機械的 rename」と「意味を変える書き換え」の境界判断が必要になる。曖昧なケースを整備として通すと不変ログが緩む懸念がある。
- **トレードオフ / 留意点**:
  - 不変ログの追跡可能性を守るため、許可は厳に「決定を変えない機械的 rename」に限定し、決定文・理由・Alternatives の書き換えは引き続き禁止する。覆す変更は supersede へ回す。

## Alternatives Considered

- **用語 rename を一切認めず supersede で対応する**: 表記統一のためだけに新 ADR を起票し旧 ADR を archive 退避するのは過剰で、連番ログに「決定を変えていない supersede」が混ざって履歴の意味が薄れる。却下。
- **整備例外を設けず古い表記を放置する**: Accepted な ADR だけ用語が揃わず、読み手が同一概念を別語と誤認するリスクが残る。doc ドリフトを許容することになり却下。
