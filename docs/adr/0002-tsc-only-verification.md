# 0002. 型チェックは tsc のみで行う（Zod 等の実行時バリデーションを採用しない）

## Context

pokeform は個体・パーティの整合性（覚えない技・使えない特性・存在しないフォルム・性格の up=down・ポイント合計 66 / 各能力 ≤32・同種族重複・未解禁レギュ）を機械的に弾きたい。これらの制約の大半は**構造的**で、TypeScript の型システムで自然に表現できる。実行時バリデーション（Zod 等）を併用すると、型と検証スキーマの二重管理が発生し、コーディングエージェントにとって「どちらが正か」が曖昧になる。

## Decision

検証の単一ゲートを `tsc --noEmit` に置く。YAML（個体）/ Markdown（パーティ）を codegen で `*.generated.ts` に変換し、不正は tsc が型エラーとして検出する。codegen は常に成功してファイルを吐き、判定は型に委ねる。診断可読性はブランドエラー型名（`MoveNotLearnedBy<"pikachu","surf">` 等）と、生成 TS の `// @source pikachu.yaml:12` コメント→元 YAML/MD 行へのマッピングで担保する。型レベル算術が重い「合計 66」のみ、codegen が合計を算出して `satisfies PointTotalMustBe66<computedSum>` を埋める方式で逃がす。詳細は `docs/plan/01-mvp/architecture.md`（検証の単一ゲート節）。

## Consequences

- **良い点**: 検証ロジックが型定義に一本化され、二重管理が消える。エージェントは `tsc` 一発で整合を確認できる。
- **悪い点 / コスト**: 型レベル算術（合計 66）が重く、codegen 側へ計算を逃がす工夫が要る。巨大 union の tsc 性能にも注意が要る（[ADR 0003](./0003-species-dex-pattern.md) で `XxxDex[Id]` アクセス主体にして緩和）。
- **トレードオフ / 留意点**: 診断は素の tsc では読みにくいため、ブランドエラー型 + `@source` マッピングで人間/エージェント向けに整形する。実行時の入力検証が必要な層（CLI の外部入力）は別途扱う。

## Alternatives Considered

- **Zod 等の実行時バリデーションを併用**: 実行時に値を検証できるが、型定義とスキーマの二重管理が生じ、エージェントにとって「どちらが正か」が曖昧になる。検証ゲートを `tsc` 一本に集約する方針（単一ゲート）と相反するため却下。
